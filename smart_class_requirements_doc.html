AI-Powered Smart Class System — Project Requirements Document

Prepared for senior developer handover  |  Date: March 2026

1. Project overview

This document outlines all technical requirements, hardware specifications, software module details, and integration procedures for the AI-Powered Smart Class System. The system is designed to assist blind and deaf users by providing real-time object detection, facial recognition, speech processing, and sign language translation. All heavy computation is offloaded to a laptop via GPU processing. The Raspberry Pi unit serves solely as the data collection and transmission device.

2. System architecture

The system is composed of three primary physical units:

  • Smart class unit — the wearable or mounted device housing the camera, microphone, and speaker, with a connector for smartphone integration.
  • Transmitter — responsible for relaying sensor data from the smart class unit to the laptop for processing.
  • Laptop (processing unit) — performs all AI inference and computation. This is the primary processing hub. The Raspberry Pi does not perform any model inference.
Critical note: All AI model processing, including object detection, facial recognition, speech-to-text, and text-to-speech, must be executed on the laptop. The Raspberry Pi is a data collection node only.

3. Hardware specifications

3.1 Raspberry Pi 4 Model B

The Raspberry Pi 4 Model B is the provided edge device. Since all processing is done on the laptop via the transmitter, the Pi functions as a data capture and forwarding unit only. No AI models should be loaded or executed on the Pi itself.

3.2 Camera module

A 5MP Raspberry Pi-compatible camera module is provided. Requirements for the camera are as follows:

  • Field of view: 120 degrees minimum
  • Autofocus: required if available within provided module; if not supported by current module, this requirement must be flagged to the senior for hardware substitution or workaround
  • Resolution and frame rate must be sufficient to support real-time object detection at usable FPS (see FPS counter requirement in section 5.6)

3.3 Audio hardware

  • USB lab microphone — for voice/audio capture
  • USB sound card adapter — for audio output routing
  • Wired earphones or speaker — for delivering audio output to the user

3.4 Power system

  • 10,000 mAh power bank — primary portable power source
  • USB cable — for standard charging of the power bank
  • Solar panel integration — a solar panel must be connected to the power bank to allow solar charging as an alternative charging method. Both solar charging and standard USB charging must be supported. Simultaneous charging from both sources is not required; either method independently must work.

3.5 Storage

A 32GB microSD card is provided. However, since all processing is performed on the laptop and not on the Raspberry Pi, the microSD card usage may be minimal. The senior developer must assess whether the microSD card is required for OS booting of the Pi only, or if any local storage is needed for buffering. This must be clarified during integration planning.

3.6 Thermal management

A heatsink and small cooling fan are included in the provided hardware. Since the Raspberry Pi 4 is not performing any intensive AI inference, thermal load on the Pi is expected to be low. However, an enclosure fan and appropriate enclosure ventilation are still required to prevent passive heat buildup within the housing. The senior must determine whether the heatsink and active fan are necessary given the reduced processing load on the Pi.

3.7 Enclosure and connectors

  • Physical enclosure — required to house all components securely
  • Enclosure fan — required for ventilation within the enclosure
  • Wire connectors — required for all internal component connections
  • Mounting hardware — required for secure physical assembly

4. Processing and GPU requirements

All AI inference must run on the laptop using GPU acceleration. The following requirements apply:

  • The laptop must have a dedicated NVIDIA GPU
  • CUDA must be installed and configured on the laptop
  • All models must be deployed with CUDA support enabled to ensure real-time performance
  • The senior developer must verify CUDA version compatibility with all selected models and frameworks before deployment

5. Software modules

5.1 Object detection

  • Model: YOLOv8x with FP16 (half-precision) inference
  • This specific model variant has been validated in prior projects and is the required implementation
  • Must run with CUDA acceleration on the laptop GPU
  • Must support real-time detection with FPS counter output (see section 5.6)

5.2 Facial recognition

  • Any modern, actively maintained facial recognition model is acceptable (e.g., DeepFace, InsightFace, FaceNet, or similar)
  • The model must support bounding box detection and identity tagging
  • Older or deprecated models must not be used

Face tagging workflow (add face):

  • Step 1: The user provides a name before the detection begins
  • Step 2: The facial recognition module is activated and begins detecting faces in the camera feed
  • Step 3: The user clicks on the bounding box of the face they wish to tag
  • Step 4: The system continues to observe that face continuously across frames
  • Step 5: After approximately 50 to 60 continuous successful detections of that face, the identity is confirmed and the face is permanently tagged with the provided name
  • This process ensures accuracy by requiring sustained detection rather than a single-frame capture

5.3 Speech-to-text (STT)

  • Any modern STT model is acceptable, including locally-run models (e.g., Whisper) or API-based solutions (e.g., Gemini API or similar)
  • Must be fully functional and integratable with the chatbot orchestrator
  • Must support real-time or near-real-time transcription

5.4 Text-to-speech (TTS)

  • Any modern TTS model is acceptable, including locally-run or API-based solutions
  • Must produce clear, natural-sounding audio output
  • Must be integratable with the chatbot and individual modules

5.5 Sign language translation

This module is being developed and integrated by a separate team member. No further specification is required from this document. The senior developer should coordinate with the responsible party for API or module integration details.

5.6 FPS counter

  • A real-time FPS counter must be displayed during active detection modes
  • The counter allows the developer and user to monitor system performance
  • Must be visible in the interface overlay during runtime

5.7 Voice assistant and mode announcer

This module is being handled by a separate team member. The module must narrate which operational mode the system is currently in (e.g., "Object detection mode active", "Facial recognition mode active"). Integration coordination with the chatbot orchestrator is required.

5.8 Chatbot orchestrator

The chatbot is the central controller that ties all modules together. Its operational flow is as follows:

  • Step 1: System listens passively for the designated wake word
  • Step 2: Upon wake word detection, the system begins capturing voice input
  • Step 3: Speech is converted to text via the STT module
  • Step 4: The chatbot analyses the transcribed text to determine user intent
  • Step 5: The appropriate module is triggered (e.g., object detection, facial recognition, STT, TTS)
  • Step 6: Output from the triggered module is processed and converted to audio via the TTS module
  • Step 7: The audio response is delivered to the user through the speaker or earphones

6. Settings module

A dedicated settings interface must be included. The following parameters must be configurable:

  • Display brightness
  • Camera resolution
  • Model selection — ability to switch or check which model is currently loaded for each module
  • Additional system parameters as determined during development

7. Add object and add speech features

These features allow users to register new items or voice samples into the system. The implementation approach for these features is to be determined by the senior developer based on the architecture of the chosen models. The add face workflow is defined in section 5.2.

8. Integration and deployment notes

  • All software must run on the laptop as the primary processing device
  • CUDA must be verified and configured before any model deployment
  • The Raspberry Pi communicates with the laptop via the transmitter; the communication protocol and data streaming method are to be specified by the senior developer
  • All modules must be individually testable before full system integration
  • The microSD card use case on the Pi must be clarified (OS only vs data buffering)
  • Solar panel integration must be tested independently before being combined with the main power circuit
